Welcome to MCAS home page

Media Content Adaptive System (MCAS) offers a cloud-based software-as-a-service video and audio transcoding solution. Using a simple RESTful API is easy to connect with providers in order to stream content over the internet to a wide range of devices.

MCAS makes use of cloud platforms to achieve massive scalability and reach the maximum efficiency transcoding outputs in parallel.

Main Features

MCAS is designed to enable live or video on demand streamingusing common industry de facto standards, as is Apple HLS, Adobe RTMP or RTSP/RTP with H264 or VP8 and AAC or MP3. Coding options are highly configurable via user defined presets. Presets are configured by two abstraction layers: levels and profiles. Levels are codec agnostic and they make reference to quality (aspect ratio, maximum bitrate, compression level, etc.). Profiles are codec and protocol dependent (HLS, RTMP, MP4, etc.). So a preset is a collection of profiles and levels. Presets, profiles and levels are easily configured in an external SQL database. MCAS workflow has been simplified to the minimal expression by the next RESTful methods:

  • addTranco: given a media source (RTSP/RTP or media file), a preset and destination, starts a transcodification job.
  • addCustomTransco: given a media source (RTSP/RTP or media file), a destination, a profile and level, starts a transcodification job.
  • cancelTransco: given a transcodification id cancels the correspondent job if it’s still present in system.
  • getOptions: it returns available profiles, levels and presets.

MCAS Presets, Profiles and Levels

Transcoding configuration is stored in a database which can be external and accessible by the MCAS system or may be internal, as an embedded database. It supports any Hibernate 4.0 compatible database server. Presets, levels and profiles are defined in the database as it follows.

Presets

Presets are a collection of profiles and levels. Each preset may include one or more profiles and each profiles is related to one or more levels. Relevant profile parameters are:

  • Name (string): name of the preset, must be unique.
  • Live (boolean): a flag to notify whether it is a live stream (1) or a video on demand stream (0).
  • Timeout (integer): timeout for the transcodification process expressed in seconds.
  • Profiles: list of profiles for that preset.
  • Levels: list of levels for a profile. Basically defining a preset consists of deciding which formats (one per profile) and which qualities (one per level) are going to be used on the output. For example, a common configuration for live streaming would be to enable HLS and RTMP profiles with high and low quality level for each profile. With this configuration the transcoding would be available in two different bitrates for each stream format (HLS and RTMP). As an example it could be written in XML as it follows:
<preset name="example">
    <timeout>86400</timeout>
    <live>false</live>
    <profiles>
        <profile name="HLS">
            <levels>
                <level name="high"/>
                <level name="low"/>
            </levels>
        </profile>
        <profile name="RTMP">
            <levels>
                <level name="high"/>
                <level name="low"/>
            </levels>
        </profile>
    </profiles>
</preset>
Profiles

A profile defines the codec and protocol for streaming (HLS, RTMP, MP4 or WebM). Relevant parameters are:

  • Name (string): name of the profile.
  • Format (string): the output format/container (mp4, webm, m3u8, flv, etc.)
  • Video Codec (string): the video codec that is going to be used (mostly libx264 or libvpx).
  • Audio Codec (string): the audio codec that is going to be used (libfaac, libvorbis, libmp3lame).
  • Additional Flags (string): additional flags that may be needed depending on codec (for example “-profile:v baseline” if we are transcoding in h264 for mobile devices). As an example it could be written in XML as it follows:
<profile name="H264">
    <format>mp4</format>
    <vcodec>libx264</vcodec>
    <acodec>libfaac</acodec>
    <additionalFlags>-profile:v baseline</additionalFlags>
</profile>
Levels

A level defines some quality aspects of the desired output, like compression level, maximum bitrate or size. Relevant parameters are:

  • Name (string): name of the profile.
  • Width (integer): desired width of the output, the height is calculated in consequence in order to keep original aspect ratio. Width value of -1 is used to keep original size.
  • Quality (integer): it defines the compression level (Constant Rate Factor), it is a value from 0 to 50 where 0 means lossless compression level and 50 is the highest compression. Commonly is considered that values under 18 are considered to be visually lossless.
  • Audio bitrate (integer): it defines the constant audio bitrate, expressed in Kbits per second.
  • Maxrate (integer): it defines the maximum video rate for the output, expressed in Kbits per second. As an example it could be written in XML as it follows:
<level name="high">
    <width>-1</width>
    <quality>20</quality>
    <abitrate>128</abitrate>
    <maxRate>8000</maxRate>
</level>

Product Specification

Protocols

RTP (RFC 3550), TCP (RFC 793), UDP (RFC 768), HTTP (RFC 2616), HLS, RTPM

Video Codecs

Flash Video (FLV), H.261, H.263, H.264 / AVC / MPEG-4 AVC, MJPEG, MPEG-1/2 video, MPEG-4 part 2, VP8

Audio codecs

G.711 µ-law (PCMU), G.711 A-law (PCMA), AAC, G.722, G.726, FLAC, Speex, Vorbis, MP3, Opus

System Management

Web-based controlbased on a RESTful API toenable third-partyintegration

Additional Features

Active callback on work process update via text file or a RESTful method call. Supports external self-defined callback class for custom integration.

Operating Systems
  • Linux (Ubuntu 10.04/12.04)
  • Windows 7
